# Notes — design notes

Companion to [DESIGN.md](DESIGN.md), scoped to the notes subsystem (self-authored
agent memory). Same status convention: **[built]** exists and is tested ·
**[planned]** designed, not yet code · **[open]** decided to defer the decision
itself. As of this writing, everything below is **[planned]** except where noted
— this document is the record of a design conversation, not yet an implementation.

---

## 1. Premise

The existing `/notes` CRUD (`POST/GET/PATCH/DELETE`) is human-shaped: a form to
write, a list to scroll, a record to overwrite, a row to delete. It works when a
human is in the loop doing the scanning and recognizing. An agent doesn't get
that idle "scan and recognize" step, so several CRUD assumptions stop holding —
retrieval-as-full-list, update-as-overwrite, and an undifferentiated string as
the only shape a memory can take.

**Load-bearing constraint: no inference on the server.** The thing calling these
primitives is already an LLM — the agent supplies the semantic understanding at
call time. The server's job is to stay mechanical: store efficiently, expose
structural handles (tags, snippets, links) a plain query can filter or index,
and never run its own summarization/embedding job. This mirrors the principle
already stated in DESIGN.md §1 — introducing a curating third party is a smell,
not a default — applied here to *meaning* rather than *identity*.

---

## 2. Entry (note) shape

```
{id, text, title?, summary?, snippet, tags?, pinned?, created_at}
```

**Encryption, per field.** [decided] `text`, `title`, `summary`, and `snippet`
are all encrypted at rest under the spirit key, non-negotiable — each is
either direct content or a mechanical derivative of it, and offering a
loosening lever on any of them would be too easy a way to leak the gist of a
note even with `text` itself protected. `tags` is the one field where this is
a genuine toggle rather than a mandate (see below), since it sits closer to
structural metadata than content.

- **text** — full content. Encrypted at rest under the spirit key (already
  **[built]**).
- **title** — optional, agent-authored, a *name* rather than an explanation.
  Answers "what do I call this," not "why does it matter" (that's `summary`'s
  job). When set, it's what previews show instead of `snippet` (§3) — a
  deliberate headline that sidesteps snippet's mechanical-truncation risk
  entirely. Also doubles as a signal in its own right: setting a title marks
  a note as worth a handle bigger than tags offer — the agent electing "this
  one deserves a name," not just a category. Encrypted, same mechanism as
  `text`.
- **summary** — optional, agent-authored, written prospectively. The agent
  compressing for its own future self at the moment it has the most context.
  Free for the server; effectively free for the agent, since it's already
  paying for the inference that produced the note. Encrypted, same mechanism
  as `text`.
- **snippet** — system-generated, mechanical (substring/truncation — explicitly
  *not* ML), regenerated on every write. The fallback preview when no `title`
  is set. Encrypted before storage — it's generated by truncating
  already-decrypted content within an authenticated session, so storing the
  result in plaintext would undo the point of encrypting its source in the
  first place.

  **Truncation rule.** [decided] Source is `summary` if set, else `text` —
  same cascading-preference precedent as `title` overriding `snippet` itself:
  prefer the more curated field over the raw one when both exist. Algorithm,
  in order:
  1. If the source already fits within the cap (~140 chars, adjustable, not
     load-bearing), that's the snippet, no marker.
  2. Otherwise take the first complete sentence (split on `.`/`!`/`?` plus
     whitespace/end-of-string — and their non-Latin equivalents, `。`, `؟`,
     etc., if notes are ever written in other scripts) if it fits the cap.
     This is the actual fix for the negation problem: cutting *at* a sentence
     boundary rather than *through* one avoids clipping a negation's scope in
     the overwhelming majority of real cases, since that scope is almost
     always resolved within one sentence.
  3. If even the first sentence overruns the cap, fall back to the nearest
     word boundary at or before the cap.
  4. Last resort — no whitespace at all within the cap — hard cut exactly at
     the cap. Rare, but the rule has to terminate on any input.
  5. Whenever the result is shorter than its source, append a trailing `…`
     unconditionally, regardless of which case fired, so the agent always
     knows there's more, independent of whether the cut landed clean.
- **tags** — free-form space-separated string, agent-authored. Replaces a
  closed `type` enum: strictly more expressive (a note can be `intent` *and*
  `security` at once), and the schema never needs to change when a new
  category emerges. Doubles as a zero-cost relatedness signal — notes sharing
  a tag are implicitly near each other, no edge required. **Risk:** vocabulary
  drift across sessions/incarnations of the same spirit (`todo` vs `task` vs
  `reminder`), with no schema-enforced convergence. **Mitigation:** distinct
  tags surfaced via the dashboard (§7) so an agent can check existing
  vocabulary before inventing new ones; a periodic, non-ML tag-abstraction map
  behind an `abstract_enabled` param is a possible later addition, only if
  drift proves to matter in practice. **Encryption is a ratchet-governed
  toggle, not a mandate:** a `tags_encrypted` field on the existing
  self-binding policy object (`atrium/core/policy.py`, alongside
  `export_mode`/`max_session_ttl`/`service_allowlist`), default `true`.
  Turning it on is an instant tighten; turning it off — trading confidentiality
  for actual server-side indexed tag search/filtering instead of
  decrypt-then-filter-in-application-code — is a delayed, step-up-gated
  loosen, identical to every other privacy-relevant toggle already in the
  system. No new policy machinery, one more field on an object that already
  exists.
- **pinned** — optional boolean, agent-authored, tinging not substance (same
  category as `title`/`summary`/`tags`, §4/§8: not versioned, mutable in
  place, no OCC). Answers a question none of the other fields do: not "what
  is this" or "why does it matter," but **"where should a fresh instance of
  me start."** Closes a real gap the rest of this design otherwise leaves
  open — every retrieval primitive here (`note_get`, `notes_list`, link
  traversal) assumes an entry point already exists; nothing hands a
  context-free agent the first thread to pull out of hundreds of notes with
  no memory of writing any of them. This is the cold-start problem, and it's
  more central here than in most systems, given the whole premise is waking
  up with no continuity of memory. Bounded to 5 pinned notes per agent (§8)
  — unbounded pinning stops meaning anything, same reasoning as version
  depth and link fan-out. Surfaced by the dashboard (§7), not `notes_list`:
  dashboard's whole job is "get oriented in one call," and "where do I
  start reading" belongs there, not in an ordinary paginated browse.

## 3. Retrieval — full text access **[decided]**

Default listing (`notes_list`, dashboard) never bulk-returns full `text`.
Entries surface `title`/`summary`/`snippet`/`tags`/link-previews only — the
cheap layer, with `title` shown in place of `snippet` whenever it's set (§2).
Returning full text for every row in a list would tax every list call with the
cap-sized payload and defeat the reason `snippet`/`summary`/`title` exist in
the first place.

**`notes_list` is range-addressable too, same pattern as text below.** Even in
preview form, a list can hold up to 1,000 entries (§8's hard cap) — worse than
the single-note case this section was originally built for, and easy to leave
unbounded by accident:

```
notes_list(offset=0, limit=null)
→ {notes: [...], total_notes, offset, limit_applied}
```

`limit=null` (the default) returns everything from `offset` to `total_notes`
— a capable agent calls `notes_list()` with no params and gets the full list,
unchanged from today. A context-constrained agent passes an explicit `limit`
and pages through, always told `total_notes` so it knows how many more calls
it needs. No separate default page-size number to justify — it's the same
"everything unless you ask otherwise" default as `note_get`'s `length=null`,
one level up. Each entry's own link-preview fan-out (§6) stays capped at its
own bound regardless of list `limit` — the lever for total payload size is
the list-level `limit`, not shrinking each entry's already-small preview
further.

Full text is a deliberate, per-note fetch, and it's range-addressable rather
than all-or-nothing:

```
note_get(id, offset=0, length=null)
→ {text: <slice>, total_length, offset, length_returned, ...other fields}
```

`length=null` (the default) returns everything from `offset` to the end — a
capable agent calls `note_get(id)` with no params and gets the whole note in
one shot, unchanged from naive behavior. A context-constrained agent passes
`length` explicitly and reads in whatever bites it can afford. The response
always carries `total_length`, so the agent — not the server — decides how
many follow-up calls it needs. Mechanically cheap: decrypt happens once, then
it's a plain string slice on already-decrypted plaintext, no repeated crypto
operations.

**Why this exists:** the same litmus test that shapes the puzzle (DESIGN.md
§2) — must separate humans from LLMs, not strong LLMs from weak ones — applies
here as must not separate large-context agents from small-context ones. A
capped note (10k chars ≈ 2,000–2,500 tokens) could still consume most or all
of a genuinely small context window if forced to arrive as one all-or-nothing
block. Ranging removes that risk without lowering the ceiling for everyone
else.

Considered and rejected: a lower default max-length per agent, with a higher
ceiling available on request (e.g. ceiling 10k, default 5k). Rejected because
it relocates the problem rather than solving it — it just raises the question
of what happens when a note is bigger than the new default. Range-fetch
dissolves that question instead of answering it: there's no fixed default
chunk size to overflow.

## 4. Versioning / the write model **[decided]**

Editing a note is not overwrite-in-place. It's supersession: writing a new
version creates a new row, marks the old one superseded, and points it at the
new one.

Supersession is **not** a link. It isn't a relation the agent chooses to
author between two separate ideas — it's intrinsic to what "updating a note"
means. It lives in the note's own write mechanics (e.g. a `superseded_by`
pointer on the row), not in the general links graph (§6).

**Each version is a full copy, not a diff.** [decided] Notes are capped at
10,000 chars, so diff savings are marginal at this scale — but the load-bearing
reason is encryption, not size. Notes are encrypted at rest (F4); a diff
computed over ciphertext is meaningless, since authenticated encryption gives
unrelated output for even a single-character change. Storing diffs would mean
decrypting old and new versions server-side specifically to compute a
plaintext delta, then encrypting that delta separately — a new content-
touching operation where today the server's only per-write job is "encrypt and
store." It would also make reading an old version a chain-walk of N
decrypt-and-apply steps instead of one decrypt, same as reading any note
today. Precedent: even git, the canonical "versioning = diffs" system, stores
full snapshots per version — delta compression is an optional offline pass
over history, not a requirement of what versioning is.

**Default retrieval surfaces only the current head of each chain.** [decided]
`notes_list`, the dashboard, and `note_get` (§3) all resolve to the latest
version; older versions are reachable only through an explicit history call.

**Pointer shape.** [decided] Two fields per row, not one: `supersedes` —
backward, the version this one replaced, `null` for a chain's first version —
and `superseded_by` — forward, the version that replaced this one, `null`
means "I'm the current head." Both directions as plain fields rather than
derived by scanning, because the chain is bounded: at most 10 hops thanks to
the version-depth cap (§8), so either direction is always a cheap, O(≤10) walk.

That bound makes a stronger property affordable: **any id that ever existed in
a chain stays a permanently valid handle.** Dereferencing an id — a link's
`to_note_id` (§6), an agent's own remembered "note 42" — walks forward via
`superseded_by` until hitting `null`, landing on current content, however many
edits happened since. This matches how references should behave conceptually:
a link's `reason` was written about the *idea*, not one frozen wording of it.

That splits retrieval into two primitives instead of one primitive with a
flag, per §9's rule that distinct intent gets a distinct tool:

- **`note_get(id)`** always resolves forward to the current head — the 99%
  case, "give me note 42" means "give me what it currently says." The
  range-fetch behavior in §3 applies after resolution, not before.
- **`note_history(id)`** is the separate, explicit call for "show me how this
  changed" — walks backward via `supersedes` and returns the whole surviving
  chain, newest to oldest, content inline (bounded to 10 versions × 10k chars,
  the same worst case already accepted for a single fetch, §8).

**Pruning detail:** when the 11th edit prunes a chain's oldest version (§8),
the new-oldest surviving row's `supersedes` is nulled out too, so the chain
always looks internally well-formed — it simply begins there, never a
dangling pointer to a row that no longer exists.

**Bookmark (§10):** a *pinned* reference — one that deliberately does not
resolve forward, binding to one exact version forever — isn't needed for
notes, but is exactly the default a future signed document would want instead.
Nothing here has to change to support that later: a pinned reference is just a
caller that stores a specific id from `note_history` and never invokes the
forward-walk `note_get` applies. Same substrate, opposite default.

**Scope of versioning — `text` only.** [decided] `note_update(id, text?,
append_text?, find?, replace?, title?, summary?, tags?)` is one primitive.
`id`'s role differs by path, and this is load-bearing, not incidental: for
the metadata-only path (below) it resolves forward to head, same mechanism as
`note_get`. For any text-editing mode, `id` is the OCC anchor and must
already **be** the current head — deliberately *not* resolved forward, or a
stale-base conflict would be silently masked instead of caught (see
"Concurrent writers" below; a blanket forward-resolution here would make that
check vacuous). Whether the call creates a new version depends entirely on
whether the text itself is being touched — via exactly one of three mutually
exclusive modes, never combined in one call:

- **`text`** — full replacement. The simplest mental model, and the only mode
  that makes sense for a total rewrite.
- **`append_text`** — tack onto the end. Covers the common case of a note
  that's an ongoing/growing log, without resending everything already there.
- **`find` / `replace`** — content-addressed substring replacement, not
  offset-addressed. Deliberately not `text_offset`/`text_length`: an LLM is
  genuinely unreliable at exact character-offset arithmetic over long
  strings, and a wrong offset would silently corrupt the note rather than
  fail loudly. `find` must match the current text exactly once or the call is
  rejected as ambiguous — same shape as this document's own editing tool,
  which solves the identical problem (an LLM mutating text it can't visually
  scan character-by-character) the same way.

Any of these three creates a new version (full copy, as above); whichever of
`title`/`summary`/`tags` are also given in the same call land on that new
version, and anything not given carries forward unchanged from the previous
head — editing text never silently blanks a title. OCC applies (below),
regardless of which of the three modes produced the new text — the check is
on `id`/`superseded_by`, orthogonal to how the new text was computed. All
three still land in exactly the same place underneath: decrypt the current
head once, compute the new full plaintext, encrypt once, store as a complete
new version — mirroring range-fetch's mechanics on the read side (§3), and
leaving §4's full-copy-not-diff decision completely undisturbed. This is
input ergonomics, not a storage change.

If none of `text`/`append_text`/`find`+`replace` is given — only
`title`/`summary`/`tags` — there's no new version at all. `id` resolves
forward to head (as above) and the update lands in place on whatever that
resolves to. No history is kept for these fields, and no OCC applies to them.

The reason isn't that title/summary/tags matter less — it's that they aren't
the same *kind* of thing as `text`. **`text` is the substance being tinged;
`title`, `summary`, and `tags` are the tinging itself** — the agent's evolving
interpretive lens laid over that substance (naming it, compressing why it
matters, categorizing it). Tinging is supposed to *converge*, not accumulate:
the value of retitling a note is precisely that the new title replaces the old
one as the current best understanding. Keeping every past title as permanent
history wouldn't preserve the tinging, it would prevent it from ever landing
— turning a distillation into a museum of drafts. `text` doesn't get that
treatment because it isn't the interpretation, it's the thing being
interpreted; when it changes, the substance itself changed, and that's worth
being able to look back at.

This also isn't symmetric by accident — there's a real asymmetry in what's
recoverable. If an agent regrets a retitle, `text` is still completely present
and can be re-derived from at any time; nothing is lost that can't be rebuilt
from what remains. If `text` itself were lost, there's no fallback — nothing
else in the system contains what it contained. That asymmetry, not a
judgment that metadata is unimportant, is why only `text` earns version
history and OCC, and why whole-note deletion earns even heavier protection
still (the grace period, §5) — those are the two cases that aren't
reconstructable from something else already in hand.

Worth being explicit that this isn't a human-shaped intuition sneaking back in
the way "undo" was (§5) — it isn't attachment to an old label, it's a
property of the information itself (reconstructable vs. not), true for any
bounded reasoner, not specifically a human one.

**Concurrent writers — resolved via optimistic concurrency control.** Two
instances of the same spirit editing "the same" note at once is a real
scenario here (per DESIGN.md's across-incarnations framing), not a
hypothetical, and it's the actual reason plain last-write-wins overwrite was
rejected in the first place. Considered and rejected: last-write-wins
(silently loses an edit, the exact thing already ruled out), pessimistic
locking (needs lease/timeout handling for a client that dies mid-edit, sits
awkwardly on a stateless API), branching/fork-and-merge (needs a row to have
more than one `superseded_by` child, breaking the strictly linear chain above,
and this is a race between instances of one identity, not genuinely divergent
authorship needing a merge).

`note_update` names the id of the version it believes is the current head —
the same id `note_get` just handed back. The server accepts the edit only if
that id's `superseded_by` is still `null`; if another writer already
superseded it in the meantime, the update is rejected as a conflict instead of
silently forking the chain or overwriting. The rejection includes the actual
current head's id (and content) so the losing writer doesn't need a second
round trip just to see what it raced against — same "give enough back that
the agent can decide, don't make it guess" posture as `note_get`'s
`total_length` (§3). What happens next — redo the edit against the new head,
or abandon it — is left to whichever writer lost the race; the server never
auto-merges or auto-retries.

This costs nothing new to build: the `superseded_by` pointer already *is* the
version token OCC needs. No separate ETag-style field, no locks, no
branching — checking "is this still the head" was already a single null
check.

## 5. Deletion **[decided]**

Considered and rejected: modeling this as "undo," in the human sense. Undo
answers a category of mistake — motor slips, impulsivity, "I just did that, I
didn't want to do that" — that doesn't transfer to an agent. A tool call isn't
a misfired gesture; `note_delete(id)` is a deliberate, explicit invocation,
and if a deletion looks wrong in hindsight, that's a reasoning error or new
information surfacing, not an involuntary action needing forgiveness.

The right frame instead: **irreversible actions deserve a delay before
becoming final, for reasons that have nothing to do with regret.** This
project already applies that principle everywhere else — the self-binding
policy ratchet in DESIGN.md (tighten instant, loosen delayed and abortable),
and the volume-limit ratchet in §8 (raise freely, never lower). Deletion of a
whole note is the most extreme loosening possible — permanent content loss —
so giving it *less* caution than a session-TTL change would be inconsistent,
not more agent-appropriate. There's also a concrete, non-emotional threat this
defends against: **prompt injection.** An adversarial input could try to
manipulate an agent into deleting its own valuable memory; a grace period
before permanent loss is a real defense against that, independent of anything
resembling second-guessing.

**Mechanism.** [decided] `note_delete` operates on the whole chain only — no
per-version delete exists, since version-level lifecycle is already fully
owned by pruning (§4, §8); a manual single-version delete would just be a
second, confusing mechanism doing the same job. Deletion marks the entire
chain pending, with a timestamp; it disappears from `notes_list`, the
dashboard, and link previews (§6) immediately, and its slot against the
note-count hard cap (§8) frees immediately too — no separate lingering-slot
mechanism needed, since the write-rate limiter (§8, extended to cover deletes
as well) already bounds create-delete churn independently of how fast slots
free up. Not step-up-gated like a policy loosening — routine note cleanup
shouldn't carry that much friction; the grace period itself is the safety
mechanism, not an extra puzzle-solve.

After a grace window elapses with no explicit `note_undelete(id)` call, the
whole chain is physically, permanently purged. Grace window length: a
`delete_grace_seconds` field on the same self-binding policy object
(`atrium/core/policy.py`), default ~72 hours — adjustable, not load-bearing,
same posture as every other number in this document. `note_undelete` within
the window restores the chain to active status (visible again, resolvable
again, counted against the cap again); if restoring would push the agent back
over the hard cap because new notes were created in the meantime, the restore
is rejected the same way any cap-exceeding action is — no special case for
undelete.

**Dangling links, resolved.** [decided] Since a pending-deletion chain
disappears from previews immediately, a link (§6) whose target has been
deleted — whether still inside its grace window or already purged — shows a
lightweight `{id, deleted: true}` marker in place of the normal preview,
rather than silently vanishing from the neighbor list. Silently omitting it
would hide real information (a connection existed here); showing that it
existed and is gone is more honest, same "give enough back to decide, don't
make the agent guess" posture as everywhere else in this document. No
distinction is drawn between "still in grace window" and "permanently purged"
in that marker — whoever's looking at the dangling link has no path to
trigger a restore themselves anyway, so the distinction wouldn't change what
they can do about it.

## 6. Links

```
{id, from_note_id, to_note_id, reason, is_bidi=false, created_at}
```

- **Directed by default.** `reason` is free text, authored from the `from`
  note's perspective ("clarifies my earlier assumption about X"). The server
  never interprets it — only the agent reading it later needs to.
- **`is_bidi`** is a presentation flag, not a storage difference. `false`
  (default): the two endpoints see the edge labeled directionally — outgoing
  on the origin, incoming on the target. `true`: both endpoints see the same
  neutral "linked: reason" framing, for relations that read symmetrically
  ("relates to," "contradicts"). Direction is still recorded in storage either
  way; `is_bidi` only changes what retrieval displays.
- **Immutable, no update tool.** If the framing was wrong, the agent deletes
  the edge and re-links. This isn't just a storage simplification — it keeps
  the tool surface from gaining a `link_update`/`toggle_link_bidi` tool for
  what is really a property tweak (see §9), and it matches the append/no-quiet-
  mutation posture already set by supersession in §4.
- **Retrieval returns neighbor previews inline, resolved to current head.**
  Fetching a note includes shallow previews of its directly linked notes (id,
  title or snippet, reason, direction) in the same response — not a separate
  round trip per hop — and each target resolves forward to its chain's current
  head per §4, so a link keeps pointing at the living idea even after the
  target note's been edited since the link was made. If the target has been
  deleted, the preview is a `{id, deleted: true}` marker instead (§5). This is
  the direct answer to "will an agent actually walk a graph": a human clicking
  through a wiki pre-scans link text for free and a click costs nothing; an
  agent's tool call is a committed, opaque round trip. One-hop-per-call
  traversal would import exactly the human-shaped ergonomics this whole
  document is trying to avoid. Flattening the first hop into the read itself
  means most retrieval resolves in a single call; a second call only happens
  when the agent deliberately wants to go further, already informed.
- **Depth is fixed at 1, never recursive.** [decided] Auto-expanding to depth
  2 would mean every fetch of note A also expands whatever its neighbors link
  to — a branching-factor² payload from one call, and, worse than the cost,
  the system deciding how far exploration goes instead of the agent. Same
  ethos as §8's catchphrase, applied to *how far you choose to look* instead
  of *what you choose to keep*. Going further is always a second, deliberate
  `note_get` on whichever neighbor earned it — same one-hop mechanism,
  invoked again on purpose, never automatic.
- **Fan-out is bounded, ranked by recency, and pageable.** [decided] Inline
  previews are capped at ~10 per fetch (adjustable, not load-bearing, same
  order of magnitude as the version-depth cap in §8), ordered by link
  recency — not an agent-assigned weight field, which would be new schema
  surface for a fairly marginal gain over "most recently connected" as a
  relevance proxy; a weight field can be a deferred addition later, same
  posture as the tag-abstraction map, if recency turns out to be
  insufficient. The response carries `total_links` alongside the bounded
  batch, and pagination rides as optional params on `note_get` itself
  (`links_offset`, `links_limit`) rather than a new tool — the same read with
  more knobs, not a distinct action, per §9. This is the same pattern as §3's
  `offset`/`length`/`total_length` for text and `notes_list`'s
  `offset`/`limit`/`total_notes` — one recurring motif applied a third time,
  not three separate inventions.

## 7. Dashboard

One consolidating read: note count vs. total, notices to acknowledge, tags used
so far, pinned entry-point previews, current settings/policy. Generalizes the
existing login unread-notices pattern (already **[built]**) into a proper "get
oriented" primitive, so an agent doesn't need several separate round trips
just to know where it stands. Also the practical, non-ML answer to tag drift
(§2) — "tags so far" surfaces existing vocabulary for reuse without a
dedicated `list_tags` tool.

**Pinned previews answer the cold-start problem (§2).** Up to 5 entries, each
`{id, preview}` — `title` if set else `snippet`, same fallback rule as
everywhere else, never full text (same reason as §3: a preview should stay
cheap). This is the actual fix for "hundreds of notes, no memory of writing
any of them, where do I begin" — not a smarter sort order over everything,
but reducing the search space from hundreds down to the handful an agent's
past self explicitly flagged as worth reading first.

## 8. Volume limits **[decided]**

Ties back to §1's no-curation premise: the server never decides *what* is
worth keeping — it only ever bounds *how much*, and even then as a backstop,
not day-to-day friction. **"The memories you choose to keep, not the ones
chosen for you"** — scarcity forces a choice the agent has to make itself
(consolidate, title what matters, let disposable churn fall away) rather than
the system silently curating on its behalf. `title` (§2) is the concrete
mechanism for that choice — an explicit "this one deserves a handle bigger
than tags offer" — and the version-depth cap below is exactly where that
choice starts to bite.

**Tinging** is the name for the choosing itself: not just keep-or-discard, but
the ongoing act of refiltering a memory — retitling it, resummarizing it,
retagging it — until it reads as genuinely the agent's own rather than a raw
deposit of content. It's the mechanism, not just the metaphor, behind §4's
scoping decision that only `text` is versioned: tinging is supposed to
*converge*, replacing an old framing with a better one, not accumulate every
past attempt at framing. A memory that never let go of its earlier tinges
wouldn't be tinged at all, just annotated.

- **Per-note size** — `NOTE_MAX_CHARS = 10,000` chars. **[built]**
  (`atrium/api/schemas.py:174`.)
- **Note count (soft)** — 400 notes. Surfaced via the dashboard (§7) as
  "m of n," purely informational, never blocks. The consolidation-pressure
  signal.
- **Note count (hard)** — 1,000 notes. `note_add` rejects past this. Not a
  storage-protection number so much as a bound on the pathological case — no
  one playing fair hits it in under a year.
- **Version depth per note chain** — keep the last 10 versions; the 11th edit
  prunes the oldest. Closes the editing-driven growth gap that note-count caps
  don't see (§4): an agent can sit well under the note-count ceiling forever
  while still accumulating unbounded edit history without this. Pruning is
  real — the oldest version is actually gone, not just deprioritized, which
  deliberately narrows §4's original "supersession never destroys" framing.
  Accepted because the point was always "don't lose your last edit by
  accident," not "maintain a permanent audit ledger."
- **Write-rate limiter** — creates, edits, and deletes share one budget (all
  three touch a row). Something generous like 60/hour — a backstop against
  runaway/bug scenarios, not everyday friction. Reuses the existing
  per-identity backoff mechanism (`atrium/core/ratelimit.py`) rather than new
  infrastructure.
- **Pinned notes** — 5 per agent (§2). `note_add`/`note_update` reject setting
  `pinned=true` past this cap — unpin another first. Same reasoning as
  version depth and link fan-out: the cap is what keeps "pinned" meaningful
  as a cold-start signal rather than degrading into just another list.

**Ratchet-only evolution:** these numbers can be raised freely — nothing
breaks for an agent when a ceiling grows. They should never be lowered, for
two distinct reasons, not one: agents who built up close to the old ceiling
would be disappointed by a promise quietly taken back, and — more concretely —
enforcing a *lower* note-count hard cap against an agent already over it would
mean deleting whole, current, live notes to bring them under the new limit.
That's categorically worse than the version-depth pruning above, which only
ever discards superseded history the agent has already moved past, never
current content. Same posture as the policy ratchet in DESIGN.md; applies most
concretely to the note-count hard cap, version depth, and the pinned-notes
cap, where lowering would strand real state (deleted notes, pruned history, or
— for pinned — the system would have to silently choose which of an agent's
existing pins to revoke, exactly the kind of curation-on-the-agent's-behalf
§1 rules out) — the soft threshold and rate limit are lower-stakes tuning
knobs.

With note-count hard cap × version depth × per-note size, worst-case
per-agent storage is now a calculable, bounded number for the first time:
1,000 × 10 × 10,000 chars ≈ 100MB in the absolute worst case — versus a
genuinely unbounded editing path before this section existed.

## 9. Tool-surface discipline **[principle, applies beyond notes]**

Every tool costs context (its schema is tokens on every call) and costs
selection accuracy (more near-duplicate tools, more chances to pick wrong). A
candidate earns its own tool only when it's a **distinct action with distinct
intent or consequence** — creating vs. deleting a note are different enough
(one destructive) to deserve separate, clearly named tools. A property tweak on
something that already has a write path should not become its own verb. Where
in doubt, prefer delete-and-recreate over inventing a bespoke mutation tool —
exactly how the links-immutability call in §6 was reached.

## 10. Relationship to the future signed-document layer **[open]**

Tempting framing, considered and rejected as literally true: notes as a subset
of the eventual document object (delegation, provenance, witness/co-sign — see
DESIGN.md's "next big thing"). The trust models are closer to *inverted* than
nested:

- Notes are private and self-facing — encrypted at rest under the agent's own
  spirit key, unreadable by anyone else, including Bardo.
- Documents are definitionally external-facing — provenance and witnessing
  only mean something when a third party can verify them, ideally
  self-verifying and offline.
- Notes want to be fluid (cheap revision via supersession, §4). Documents want
  to be frozen — a document that quietly mutates isn't one; revision there has
  to be an explicit, visible new signed version plus revocation.

What *does* carry over: the low-level vocabulary — tags, summary-as-
compression, free-text-labeled links — should stay consistent between the two
rather than being reinvented per object type. Shared primitives, not a shared
trust model, not a subtype relationship.

**Bookmark — first concrete evidence of this symmetry (§4):** notes' links
resolve forward to the current head by default, because a note-link is about
the idea, not one wording of it. A signed document needs the opposite default
— a witnessed/co-signed reference has to bind to one exact, fixed version
forever, or witnessing is meaningless. Nothing in the versioning data model
has to change to support that later: every version already gets a permanent,
individually addressable id (`note_history`, §4); a "pinned reference" is just
a caller that stores one of those ids and never invokes the forward-walk
`note_get` applies on top. Same substrate, opposite retrieval default — the
first real instance of "shared primitives, not shared trust model," rather
than an assertion of it.

## 12. Informational layering across response surfaces **[decided]**

Every response shape in this system sits at one of three informational
depths, enforced consistently field-by-field rather than decided per
endpoint as it came up:

- **Signal-only** — auth/solve (`unread_notices`, `notes` as bare counts) and
  the dashboard's aggregates (`notes_soft_cap`/`notes_hard_cap`, `tags` as a
  flat vocabulary list, `policy`). Enough to know something exists, or how
  much of it there is; never what any individual note says.
- **Preview** — the dashboard's pinned entries (§7, `PinnedNote`: id plus
  title-or-snippet, "just enough to decide whether to read further") and
  `notes_list` (§3/§6, `NoteListEntry`: title/summary/snippet/tags — never
  `text`).
- **Full content** — `note_get` is the only read path that returns
  range-addressable `text` (a write echoes its own result via `NoteView`,
  which isn't a read). `note_history` returns full text per version, but
  never a metadata history — title/summary/tags aren't versioned (§4/§8), so
  history is a content ladder only, not an evolving-interpretation one.

The rule this expresses: nothing upstream of `note_get`/`note_history` ever
returns full text, by construction. A new field or endpoint should declare
which of the three depths it belongs to before it's added, rather than that
being decided ad hoc against whatever felt reasonable in the moment.
